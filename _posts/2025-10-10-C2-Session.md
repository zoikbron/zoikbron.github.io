## Session Manager
![Design](/assets/img/posts/c2/Server%20Basics/Design.png)

This component serves as a high-level manager for all session-related functionality that is not directly tied to network operations. Its primary responsibilities include storing dynamically generated commands and defining the session API, which provides an interface for performing network-related actions.

```rust

    session_manager: Arc<MultiProtocolSessionManager>,
```

The system maintains two separate vectors for commands — one for interactive mode commands and another for global commands. 

This struct provides a high-level abstraction layer for session management, encapsulating the underlying SessionApi logic. It allows callers to interact with and manipulate sessions simplifying the overall interface.

Several additional functions were omitted from the previous overview but are worth mentioning here:

- `set_global_commands`
Stores newly generated commands from the scripting engine as global commands accessible across sessions.

- `set_interactive_commands`
Stores dynamically generated commands from the scripting engine as interactive commands specific to interactive mode.

- `send_command`
Queues a command for the active session. This function is a thin wrapper around `session_api::send_command_to_active()`.

- `send_command_to_session`
Queues a command for a specific session. It wraps `session_api::send_command_to_session()`.

- `send_coff_to_active`
Queues a COFF (Common Object File Format) task for the active session. It takes an entry point and optional arguments. Its a wrapper around `session_api::send_command_to_active_coff()`.
(This behavior will be described in more detail later.)

- `send_coff_to_session`
Similar to send_coff_to_active, but queues the COFF task for a specific session. It wraps `session_api::send_command_to_session_coff()`.

- `send_command_with_metadata`
Queues a command with associated metadata. Currently, it sends the command to the active session.

- `get_responses`
Returns a vector containing the responses associated with the active session.

- `wait_for_response`

Waits for a response associated with a specific command ID (or any if none is provided). It leverages a notifier that triggers when a new command arrives for the active session. Upon notification, it compares the received command ID with the expected one, retrieves the responses via `get_responses()`, and filters them by the matching command ID.

- `enter_shell_mode`

This function manages the transition into an interactive shell context for the active session, handling I/O streams and synchronization with shell channels.

```rust
pub struct ShellIo {
    session_manager: Arc<MultiProtocolSessionManager>,
    pub session_id: String,
    pub shell_id: String,
    pub rx: Option<mpsc::Receiver<Arc<ShellChunk>>>,
}
```
The `ShellIo` structure encapsulates the communication interface for an interactive shell within a session. It contains a receiver for `ShellChunk` objects, along with the associated `session_id` and `shell_id`. Additionally, it provides a set of wrapper methods around the `SessionApi` that facilitate sending standard input (stdin), receiving standard output (stdout), and closing the shell.

A `ShellIo` instance can be provided explicitly or left as None. Its attempts to attach to an existing shell channel or create a new one if no `ShellIo` was passed. This behavior also aligns with future command-parsing support through `clap`.

The core logic is straightforward:

If no `ShellIo` is provided, the function invokes `session_api::open_shell_on_session()`, which creates and initializes a new shell channel. The SessionApi delegates the underlying creation logic to the protocol-specific session manager.

If a `ShellIo` is provided, the function simply attaches to the existing shell channel, reusing the established communication context.

This design provides a unified mechanism for managing interactive shell sessions, supporting both new shell creation and reattachment workflows.

Some example are:

`send_command_to_session()` stores the provided command type and payload, retrieves the next command ID, creates a UniversalCommandRequest structure, and queues it.

On the other hand, `send_command_to_active_coff()` retrieves the active session, obtains the next command ID, explicitly sets `coff` as the command type, uses the binary blob as the payload, attaches metadata (such as the entry point and arguments), and then queues it.

In essence, these two functions demonstrate how different functionalities can be built on top of the same core logic with small, purpose-specific adjustments.

## MultiProtocol

The initial design works well for a single-console CLI environment. However, future iterations may require adjustments to support more flexible, multi-console access. For now, this implementation offers a solid foundation.

This struct encapsulates all data and components associated with session management. Its functions primarily operate by modifying its internal state, serving as the foundation for many of the SessionApi calls that define the expected behavior through calling many fuctions defined here.

Additionally, all network-related services interact with this struct, whether activating a session, initializing a new transfer session, or storing responses corresponding to a particular session. 

The `active_session_id` primarily serves as a convenience field used when opening an interactive console for a session. It provides a simple way to reference the currently active session without needing to pass the session ID explicitly to each function.

## Transfer
```rust
pub struct TransferManager {
    next_id: AtomicU16,
    sessions: Arc<RwLock<HashMap<u16, TransferSession>>>,
    storage_dir: PathBuf,
}
```
The `TransferManager` is responsible for orchestrating all file transfers between the implant and the server. It maintains the storage paths, transfer IDs, and active TransferSession instances. Acting as both a controller and an abstraction layer for individual TransferSession objects, it centralizes the logic for handling uploads and downloads. (for now only download is support)

The TransferManager exposes three primary functions for managing transfers:

`handle_start`
This function verifies the existence of the session and initializes a new TransferSession. The session is stored in the internal hashmap, and a response containing a unique transfer ID is returned to the implant. This ID allows the server to associate all subsequent data chunks with the correct transfer session.

`handle_chunk`
Retrieves a mutable reference to a TransferSession and passes the incoming data chunk along with the current offset. This ensures that the session continues exactly from where it left off.

`handle_complete`
Finalizes the transfer by removing the TransferSession from the internal storage. Performs any necessary checks and cleanup via the session’s finalize method to ensure the integrity and completeness of the transfer.
```rust
  pub async fn handle_start(&self, start: Start, session_manager: Arc<MultiProtocolSessionManager>, implant_id: &str) -> Result<ResponseStart, TransferError> {
        let info = session_manager.get_session_arc(implant_id).await.unwrap();
        let info = info.get_session_info().await;
        
        if start.implant_id != info.id{
            return Err(TransferError::ImplantMismatch);
        }

        let transfer_id = self.gen_id();
        let session = TransferSession::new(
            transfer_id,
            start.implant_id,
            start.filename,
            start.total_size,
            start.expected_sha256,
            &self.storage_dir,
        )
        .await?;

        let mut w = self.sessions.write().await;
        w.insert(transfer_id, session);
        Ok(ResponseStart { transfer_id })
    }

    pub async fn handle_chunk(&self, chunk: Chunk) -> Result<(), TransferError> {
        let mut sessions = self.sessions.write().await;
        let session = sessions.get_mut(&chunk.transfer_id).ok_or(TransferError::NotFound)?;
        session.write_chunk(chunk.offset, &chunk.data).await?;
        Ok(())
    }

    pub async fn handle_complete(&self, complete: Complete) -> Result<PathBuf, TransferError> {
        let mut sessions = self.sessions.write().await;
        let session = sessions.remove(&complete.transfer_id).ok_or(TransferError::NotFound)?;
        session.finalize().await
    }
```


### TransferSession
This component is responsible for managing data chunks, handling writes and updates, tracking byte ranges, and finalizing the transfer streams, whether the operation completes successfully or is aborted.

```rust
pub struct TransferSession {
    pub transfer_id: u16,
    pub implant_id: String,
    pub filename: String,
    pub total_size: Option<u64>,
    pub expected_sha256: Option<[u8; 32]>,
    pub progress_bar: ProgressBar,
    file_path: PathBuf,
    file: tokio::fs::File,
    // Track received ranges and bytes
    range_set: RangeSet,
    last_seen: Instant,

    // aborted flag + reason (if any)
    aborted: Option<String>,
}
```

The non-public fields are initialized in the constructor based on the storage directory provided by the TransferManager and a temporary filename. A progress bar is also created using the `indicatif` crate to track download or upload progress. In future versions, checksum verification may be added to ensure data integrity.

This struct exposes two primary functions:

`write_chunk()` – Handles writing incoming data chunks to the stream and updating progress.

```rust
    pub async fn write_chunk(&mut self, offset: u64, data: &[u8]) -> Result<(), TransferError> {
        if self.aborted.is_some() {
            return Err(TransferError::Aborted(self.aborted.clone().unwrap()));
        }

        let len = data.len() as u64;
        if len == 0 {
            return Err(TransferError::InvalidChunk);
        }
        let end = offset.saturating_add(len);

        // Simple overlap detection: check whether [offset, end) overlaps existing ranges
        for (&s, &e) in self.range_set.ranges.iter() {
            if !(end <= s || offset >= e) {
                // overlap
                return Err(TransferError::InvalidChunk);
            }
        }

        self.file.seek(std::io::SeekFrom::Start(offset)).await?;
        self.file.write_all(data).await?;
        self.file.flush().await?;

        // Update range set & counters
        self.range_set.insert(offset, end).map_err(|_| TransferError::InvalidChunk)?;
        self.received_bytes = self.received_bytes.saturating_add(len);
        self.progress_bar.inc(data.len() as u64);
        self.last_seen = Instant::now();

        Ok(())
    }
```

`finalize()` – Completes the transfer, performing any necessary cleanup and finalization tasks.

```rust

    pub async fn finalize(&self) -> Result<PathBuf, TransferError> {
        if self.aborted.is_some() {
            return Err(TransferError::Aborted(self.aborted.as_ref().unwrap().clone()));
        }

        // If we know total size, check completeness
        if let Some(total) = self.total_size {
            if !self.range_set.is_complete(total) {
                return Err(TransferError::Incomplete);
            }
        } else {
            // If total size unknown, we can attempt to get file length from disk and verify ranges cover 0..len
            let meta = self.file.metadata().await?;
            let file_len = meta.len();
            if !self.range_set.is_complete(file_len) {
                return Err(TransferError::Incomplete);
            }
        }
        // Move file into final destination (rename). For simplicity, rename to filename in same dir
        let dest = self.file_path.with_file_name(&self.filename);
        fs::rename(&self.file_path, &dest).await?;
        self.progress_bar.finish_with_message("Transfer complete");
        Ok(dest)
    }
```
